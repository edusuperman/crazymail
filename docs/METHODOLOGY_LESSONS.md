# 测试方法论反思：Random 按钮 Bug 为何未被发现

## 问题描述
点击"Random"按钮后，页面显示 toast 通知"已生成新邮箱"，但实际邮箱地址不变。

## 根因分析
- **代码根因**：`createMailbox()` 无参数调用时，优先复用后端已有邮箱，不创建新的
- **引入原因**：修复"每次刷新创建新邮箱"问题时，复用逻辑也阻止了 Random 功能

## 测试方法论漏洞

### 漏洞1：只验证"元素存在"，未验证"功能生效"
**现状：**
```
[测试6.1] Random按钮 → 存在 ✅
```
**应该：**
```
[测试6.1] Random按钮 → 
  1. 记录当前邮箱地址 = A
  2. 点击 Random 按钮
  3. 等待 2 秒
  4. 记录当前邮箱地址 = B
  5. 验证 A ≠ B
```

### 漏洞2：未验证按钮的"预期行为"
**现状：** 测试清单中没有"点击后状态变化"的验证项
**应该：** 每个交互按钮都必须有"操作前→操作→操作后"的完整验证

### 漏洞3：E2E 测试只验证 API 层，未验证 UI 交互层
**现状：** E2E 测试通过 curl 调用 API 验证功能
**应该：** E2E 测试必须包含 UI 交互验证（点击按钮 → 检查页面状态变化）

### 漏洞4：操作手册缺少"交互行为"的具体预期
**现状：** 操作手册写了"点击随机按钮→预期：生成新的随机邮箱地址"
**应该：** 操作手册必须明确写出验证步骤：
  - "记录当前邮箱地址"
  - "点击 Random"
  - "验证邮箱地址已变化"
  - "验证后端 API 确认新邮箱存在"

## 修复方案

### 1. 代码修复
在 `createMailbox()` 中添加 `forceNew` 参数，Random 按钮调用时强制创建新邮箱。

### 2. 测试流程修复
每个交互按钮的测试必须包含：
1. **前置状态记录**：操作前记录关键状态
2. **执行操作**：点击/输入
3. **后置状态验证**：操作后验证状态变化
4. **后端一致性**：前端变化必须与后端一致

### 3. 操作手册修复
每个功能的"预期"必须包含：
- 明确的验证步骤（不只是"预期结果"）
- 具体的状态变化描述
- 可量化的验证标准
