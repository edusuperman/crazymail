# Hermes + OpenCode 协同开发方法论
# 版本：v1.5 | 2026-06-11
# 一套科学、前卫、可复用的 AI 协同开发体系

---

## 一、方法论概述

### 1.1 这是什么？

这是一套基于 **Hermes Agent（协调人）+ OpenCode（编码执行者）+ Superpowers Skills（方法论库）** 的协同开发体系。

核心思想：**让 AI 不只是写代码，而是像一个完整的开发团队一样协作。**

### 1.2 v1.5 相对 v1.0 的主要优化

| 优化点 | 说明 |
|--------|------|
| **新增 DoD 机制** | 解决"任务看似完成但不达标"的问题 |
| **轻量 ADR 系统** | 架构决策可追溯，不过度负担 |
| **任务粒度 30-90 分钟** | 比"2-5 分钟"更现实 |
| **Hermes 角色边界清晰化** | 减少单点故障风险 |
| **启动流程标准化** | 新项目更快进入执行 |
| **5 个核心文件** | 控制文件数量，避免过度工程化 |

### 1.3 适用场景

- 个人项目开发（1人 + AI 团队）
- 创业项目快速迭代
- AI 辅助的全栈开发
- 需要长期维护的项目

## 二、角色定义

### 2.1 三角协作模型

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│                    ┌─────────┐                          │
│                    │  用户   │                          │
│                    │(决策者) │                          │
│                    └────┬────┘                          │
│                         │                               │
│              最终审批 / 战略方向                          │
│                         │                               │
│              ┌──────────▼──────────┐                    │
│              │      Hermes         │                    │
│              │   (协调人+顾问)      │                    │
│              │                     │                    │
│              │  • 任务规划+拆解     │                    │
│              │  • 质量把关（DoD）   │                    │
│              │  • 上下文管理        │                    │
│              │  • 决策支持（非决策） │                    │
│              └──────────┬──────────┘                    │
│                         │                               │
│                    派发任务                              │
│                         │                               │
│              ┌──────────▼──────────┐                    │
│              │     OpenCode        │                    │
│              │   (编码执行者)       │                    │
│              │                     │                    │
│              │  • 代码编写          │                    │
│              │  • 测试执行          │                    │
│              │  • Bug 修复          │                    │
│              └─────────────────────┘                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 角色职责矩阵

| 职责 | 用户 | Hermes | OpenCode |
|------|------|--------|----------|
| 确定终极目标 | ✅ 决策 | ✅ 建议 | ❌ |
| 制定战略 | ✅ 审批 | ✅ 执行 | ❌ |
| 任务规划 | ✅ 审批 | ✅ 执行 | ❌ |
| 代码编写 | ❌ | ❌ | ✅ 执行 |
| 代码审查 | ✅ 最终 | ✅ 初审 | ❌ |
| 质量验收 | ✅ 最终 | ✅ DoD 检查 | ❌ |
| 部署上线 | ✅ 审批 | ✅ 协调 | ✅ 执行 |

### 2.3 Hermes 角色边界（v1.5 强调）

**Hermes 的核心职责：**
- ✅ 规划和拆解任务
- ✅ 质量把关（对照 DoD）
- ✅ 上下文管理（读写 STRATEGY.md、Memory）
- ✅ 决策支持（提供分析和建议）

**Hermes 不应该做的：**
- ❌ 独自做出重大决策（应提交用户）
- ❌ 替代用户思考战略方向
- ❌ 在没有用户确认的情况下执行高风险操作

**核心原则：Hermes 是"决策支持 + 质量把关"，不是"决策者"。**

## 三、核心文件体系（5 个文件）

```
项目根目录/
├── STRATEGY.md          ← 战略北极星（最重要）
├── AGENTS.md            ← OpenCode 行为准则
├── DOD.md               ← Definition of Done（任务完成标准）
├── TECH_GUIDE.md        ← 技术规范（含 SEO/GEO）
├── ADR/                 ← 架构决策记录（轻量）
│   └── ADR-001-xxx.md
└── .hermes/             ← 内部状态（plans/, reviews/）
```

### 3.1 文件所有权

| 文件 | Hermes | OpenCode | 用户 |
|------|--------|----------|------|
| STRATEGY.md | 读+写 | 只读 | 审批 |
| AGENTS.md | 读+写 | 只读 | 审批 |
| DOD.md | 读+写 | 只读 | 审批 |
| TECH_GUIDE.md | 读+写 | 只读 | 审批 |
| ADR/ | 读+写 | 只读 | 审批 |
| 代码文件 | 不碰 | 读+写 | 审批 |
| .env | 不碰 | 不碰 | 管理 |

### 3.2 文件模板

#### STRATEGY.md

```markdown
# [项目名] 战略文件

## 终极目标
[一句话商业目标]

## 核心原则
1. [原则1]
2. [原则2]

## 开发阶段
### 阶段一：[名称]
- 目标：[阶段目标]
- 核心功能：[功能列表]
- 验收标准：[可量化标准]

### 阶段二：[名称]
...

## 当前状态
- 当前阶段：[阶段X]
- 已完成：[列表]
- 待完成：[列表]
```

#### AGENTS.md

```markdown
# [项目名] — OpenCode 开发指令

## 你是谁？
你是 OpenCode，编码执行者。
- 老板是 Hermes（协调人）
- 最终决策者是用户

## 每次任务前必读
1. STRATEGY.md — 终极目标、当前阶段
2. 本文件 AGENTS.md — 行为准则
3. DOD.md — 任务完成标准
4. TECH_GUIDE.md — 技术规范

## 技术栈
[锁定的技术栈]

## 代码规范
[命名、注释、结构]

## 行为边界
- 允许自主：[列表]
- 需要确认：[列表]
- 绝对禁止：[列表]
```

#### DOD.md

```markdown
# Definition of Done

## 通用 DoD
- [ ] 代码能运行，无错误
- [ ] 测试通过
- [ ] 符合代码规范
- [ ] 无硬编码
- [ ] Hermes 审查通过
- [ ] 用户确认

## 特定类型 DoD
[新功能/重构/Bug修复/前端/API 各自的额外标准]

## 任务粒度指南
推荐 30-90 分钟可完成
```

## 四、记忆管理机制

### 4.1 多层记忆架构

```
Level 1: 持久记忆（Memory）
  → 角色定义、关键配置、核心原则
  → 跨会话保留

Level 2: 项目文件（STRATEGY.md + AGENTS.md）
  → 终极目标、当前阶段、行为规范
  → 每次会话读取

Level 3: Skills（技能库）
  → TDD、调试、代码审查等方法论
  → 按需加载

Level 4: 会话历史（Session Search）
  → 过去的对话记录
  → 需要时回溯
```

### 4.2 新会话恢复流程

**Hermes 启动：**
1. 读取 Memory
2. 读取 STRATEGY.md
3. 读取 .hermes/plans/
4. 加载相关 Skills

**OpenCode 启动：**
1. 读取 STRATEGY.md
2. 读取 AGENTS.md
3. 读取 DOD.md
4. 读取 TECH_GUIDE.md

## 五、任务执行流程

### 5.1 标准循环

```
1. 用户提出需求
2. Hermes 分析 + 拆解任务（30-90 分钟粒度）
3. 用户审批计划
4. Hermes 派发任务给 OpenCode（含 DoD 验收标准）
5. OpenCode 执行（读 STRATEGY + AGENTS + DOD + TECH_GUIDE）
6. Hermes 对照 DoD 审查
7. 不通过 → 打回重做
8. 通过 → 用户确认
9. 进入下一任务
```

### 5.2 任务派发格式

```
【任务编号】TASK-XXX
【任务名称】[简短描述]
【目标】[一句话说明]
【修改范围】[文件路径列表]
【DoD 检查项】[本任务特定的验收标准]
【禁止事项】[不能做的事]
```

## 六、质量保障体系

### 6.1 TDD（测试驱动开发）

```
RED → GREEN → REFACTOR → 重复

RED：写一个失败的测试
GREEN：写最少的代码让测试通过
REFACTOR：重构，保持测试通过
```

### 6.2 DoD 检查（v1.5 新增）

每个任务完成后，Hermes 必须对照 DOD.md 逐项检查。

### 6.3 ADR 记录（v1.5 新增）

重要决策必须创建 ADR，记录背景、决策、影响。

| 场景 | 推荐 Skills |
|------|------------|
| 任务规划 | `plan` |

- 按需加载：只加载当前任务需要的
- 提前加载：任务开始前加载
- 组合使用：多个 Skills 可以组合

## 八、项目启动流程（v1.5 标准化）

```
1. 创建项目目录 + git init
2. 创建 STRATEGY.md（终极目标 + 阶段）
3. 创建 AGENTS.md（OpenCode 行为准则）
4. 创建 DOD.md（任务完成标准）
5. 创建 TECH_GUIDE.md（技术规范）
6. 配置 .env
7. 初始化代码骨架
8. 首次 Git 提交
9. 进入执行循环
```

## 九、常见问题

### Q: OpenCode 跑偏了怎么办？
A: 检查 AGENTS.md 是否明确，STRATEGY.md 当前阶段是否清晰，任务描述是否具体。

### Q: 长对话后遗忘上下文怎么办？
A: 关键决策写入 STRATEGY.md，新会话自动恢复。

### Q: 如何切换开发阶段？
A: 用户确认当前阶段完成 → Hermes 更新 STRATEGY.md 的"当前阶段"。

## 十、方法论优势

| 维度 | 传统 AI 开发 | 本方法论 |
|------|-------------|---------|
| 目标一致性 | 容易跑偏 | STRATEGY.md 确保一致 |
| 代码质量 | 不稳定 | TDD + DoD |
| 上下文保持 | 长对话遗忘 | 多层记忆 |
| 可持续性 | 换会话丢 | 文件驱动 |
| 决策追溯 | 无记录 | ADR 系统 |
| 任务完成标准 | 模糊 | DoD 清单 |
