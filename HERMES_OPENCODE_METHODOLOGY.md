# Hermes + OpenCode 协同开发方法论
# 版本：v1.0 | 2026-06-11
# 一套科学、前卫、可复用的 AI 协同开发体系

---

## 一、方法论概述

### 1.1 这是什么？

这是一套基于 **Hermes Agent（协调人）+ OpenCode（编码执行者）+ Superpowers Skills（方法论库）** 的协同开发体系。

核心思想：**让 AI 不只是写代码，而是像一个完整的开发团队一样协作。**

### 1.2 解决什么问题？

| 传统 AI 开发的问题 | 本方法论的解决方案 |
|-------------------|-------------------|
| AI 容易跑偏，忘记目标 | STRATEGY.md 终极目标文件 |
| 长对话后遗忘上下文 | 多层记忆机制（Memory + 文件 + Skills） |
| 代码质量不稳定 | TDD + 代码审查 + 验收流程 |
| 战略与执行脱节 | 三文件分离（战略/行为/技术） |
| 多工具协作混乱 | 明确的文件所有权和信息流 |
| 项目不可持续 | Git 版本控制 + 文档驱动 |

### 1.3 适用场景

- 个人项目开发（1人 + AI 团队）
- 创业项目快速迭代
- AI 辅助的全栈开发
- 需要长期维护的项目

## 三、文件体系（核心）

### 3.1 三文件分离原则

```
项目根目录/
├── STRATEGY.md      ← 战略层：为什么做、做什么
├── AGENTS.md        ← 行为层：怎么做、规范是什么
├── [TECH_GUIDE].md  ← 技术层：具体技术实现指南
└── src/             ← 代码层：实际代码
```

**分离原则：**
- **战略文件**：定义目标和方向，很少改变
- **行为文件**：定义规范和流程，偶尔调整
- **技术文件**：定义具体实现，随技术演进
- **代码文件**：实际实现，频繁变更

### 3.2 文件所有权（关键）

| 文件 | 谁读 | 谁写 | 谁审批 | 说明 |
|------|------|------|--------|------|
| STRATEGY.md | 所有 | Hermes | 用户 | 项目北极星 |
| AGENTS.md | OpenCode | Hermes | 用户 | 行为准则 |
| TECH_GUIDE.md | OpenCode | Hermes | 用户 | 技术规范 |
| 代码文件 | OpenCode | OpenCode | 用户 | 实际实现 |
| .env | 应用 | 用户 | - | 敏感配置 |
| .hermes/plans/ | Hermes | Hermes | 用户 | 实施计划 |

**核心规则：战略文件和行为文件，OpenCode 只读不写。**

### 3.3 STRATEGY.md 模板


### 4.2 新会话恢复流程

**Hermes 启动时：**
```
1. 读取 Memory（持久记忆）
   → 角色定义、关键配置、核心原则
2. 读取 STRATEGY.md
   → 终极目标、当前阶段、待完成任务
3. 读取 .hermes/plans/
   → 当前实施计划
4. 加载相关 Skills
   → TDD、OpenCode、Supabase 等
5. 上下文恢复完成
```

**OpenCode 启动时：**
```
1. 读取 STRATEGY.md
   → 终极目标、当前阶段
2. 读取 AGENTS.md
   → 行为准则、技术栈、代码规范
3. 读取 [TECH_GUIDE].md
   → 技术实现指南
4. 准备执行任务
```

### 4.3 Memory 存储规范

**应该存入 Memory 的：**
- 角色定义（我是谁、我做什么）
- 关键配置（API Keys、数据库连接）
- 核心原则（产品优先、快速迭代）
- 用户偏好（沟通风格、技术偏好）
- 环境信息（操作系统、已安装工具）

**不应该存入 Memory 的：**
- 临时任务状态
- 具体代码片段
- 会话特定的讨论内容
- 会过期的信息

## 五、任务执行流程

### 5.1 标准流程

```
┌─────────────────────────────────────────────────────────┐
│                    任务生命周期                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 需求提出（用户）                                     │
│     ↓                                                   │
│  2. 需求分析（Hermes）                                   │
│     • 理解需求                                          │
│     • 评估可行性                                        │
│     • 确定优先级                                        │
│     ↓                                                   │
│  3. 任务规划（Hermes）                                   │
│     • 拆分为原子任务（2-5分钟/个）                       │
│     • 确定文件范围                                      │
│     • 定义验收标准                                      │
│     ↓                                                   │
│  4. 用户审批（用户）                                     │
│     • 审核计划                                          │
│     • 确认方向                                          │
│     ↓                                                   │
│  5. 任务派发（Hermes → OpenCode）                        │
│     • 结构化任务描述                                    │
│     • 包含完整上下文                                    │
│     ↓                                                   │
│  6. 代码执行（OpenCode）                                 │
│     • 读取 STRATEGY.md + AGENTS.md                      │
│     • 编写代码                                          │
│     • 运行测试                                          │
│     • 提交代码                                          │
│     ↓                                                   │
│  7. 质量验收（Hermes）                                   │
│     • 代码审查                                          │
│     • 功能测试                                          │
│     • 规范检查                                          │
│     ↓                                                   │
│  8. 用户确认（用户）                                     │
│     • 最终验收                                          │
│     • 决定下一步                                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 任务派发格式

Hermes 派发给 OpenCode 的任务必须包含：

```
【任务编号】TASK-XXX
【任务名称】[简短描述]
【目标】[一句话说明要达成什么]
【修改范围】[明确列出文件路径]
【验收标准】[明确列出通过条件]
【禁止事项】[明确列出不能做的事]
【参考文档】[相关文档路径]
```

### 5.3 验收标准

每个任务完成后，Hermes 必须检查：

| 检查项 | 标准 |
|--------|------|
| 代码能运行 | 无语法错误、无导入错误 |
| 测试通过 | 所有测试 PASS |
| 符合规范 | 遵循 AGENTS.md 中的代码规范 |
| 符合战略 | 与 STRATEGY.md 中的当前阶段对齐 |
| SEO/GEO | 遵循 SEO_GEO_GUIDE.md（如适用） |
| 无硬编码 | 密钥、URL 等走环境变量 |

## 六、质量保证体系

### 6.1 TDD（测试驱动开发）

**铁律：没有失败的测试，就不能写生产代码。**

```
RED → GREEN → REFACTOR → 重复

RED：写一个失败的测试
GREEN：写最少的代码让测试通过
REFACTOR：重构，保持测试通过
```

**违规处理：**
- 先写代码后写测试？删除代码，重来。
- 测试一次就通过？说明测试有问题，修改测试。
- 测试太复杂？说明设计有问题，简化设计。

### 6.2 代码审查

**Hermes 审查清单：**
- [ ] 代码能运行
- [ ] 测试通过
- [ ] 符合命名规范
- [ ] 无硬编码
- [ ] 有适当注释
- [ ] 符合 AGENTS.md 规范
- [ ] 符合 STRATEGY.md 当前阶段

### 6.3 阶段验收

每个阶段完成后，必须验收：

| 验收项 | 方法 |
|--------|------|
| 功能完整性 | 对照阶段目标逐项检查 |
| 代码质量 | 运行测试、代码审查 |
| 文档完整性 | STRATEGY.md、AGENTS.md 更新 |
| 可部署性 | 能否正常部署运行 |

## 七、Skills 使用指南

### 7.1 Skills 是什么？

Skills 是 Hermes 的"方法论库"——预定义的工作流程和最佳实践。

### 7.2 推荐 Skills 组合

| 场景 | 推荐 Skills | 说明 |
|------|------------|------|
| 任务规划 | `plan` | 拆分任务、制定计划 |
| 代码开发 | `opencode` + `test-driven-development` | 派发任务 + TDD |
| 数据库集成 | `supabase` | Supabase 最佳实践 |
| 代码审查 | `requesting-code-review` | 质量把关 |
| 调试问题 | `systematic-debugging` | 系统化调试 |
| GitHub 操作 | `github-pr-workflow` | PR 管理 |

### 7.3 Skills 加载原则

- **按需加载**：只加载当前任务需要的 Skills
- **提前加载**：在任务开始前加载，不是遇到问题才加载
- **组合使用**：多个 Skills 可以组合使用

## 八、项目启动流程

### 8.1 新项目初始化

```
1. 创建项目目录
2. 初始化 Git 仓库
3. 创建 STRATEGY.md（定义终极目标和阶段）
4. 创建 AGENTS.md（定义 OpenCode 行为准则）
5. 创建 [TECH_GUIDE].md（技术规范）
6. 配置 .env（环境变量）
7. 初始化代码骨架
8. 首次 Git 提交
```

### 8.2 文件清单

新项目必须包含：

```
project/
├── STRATEGY.md          ← 战略文件（必须）
├── AGENTS.md            ← OpenCode 行为准则（必须）
├── [TECH_GUIDE].md      ← 技术规范（推荐）
├── .env.example         ← 环境变量模板（推荐）
├── .gitignore           ← Git 忽略规则（必须）
├── .hermes/plans/       ← 实施计划目录（推荐）
└── src/                 ← 源代码
```

## 九、常见问题

### 9.1 OpenCode 跑偏了怎么办？

**检查：**
1. AGENTS.md 是否明确？
2. STRATEGY.md 的"当前阶段"是否清晰？
3. 任务描述是否足够具体？

**修复：**
1. 更新 AGENTS.md，增加约束
2. 在任务中明确禁止事项
3. 必要时重置 OpenCode 会话

### 9.2 长对话后遗忘上下文怎么办？

**预防：**
1. 关键决策写入 STRATEGY.md
2. 关键配置写入 Memory
3. 定期提交代码（每完成一个任务就提交）

**恢复：**
1. 开启新会话
2. 让 Hermes 读取 STRATEGY.md + Memory
3. 上下文自动恢复

### 9.3 如何切换开发阶段？

**步骤：**
1. 用户确认当前阶段已完成
2. Hermes 更新 STRATEGY.md 的"当前阶段"
3. 必要时更新 AGENTS.md 的阶段描述
4. 开始新阶段的任务

## 十、方法论优势

### 10.1 与传统 AI 开发对比

| 维度 | 传统 AI 开发 | 本方法论 |
|------|-------------|---------|
| 目标一致性 | 容易跑偏 | STRATEGY.md 确保一致 |
| 代码质量 | 不稳定 | TDD + 审查流程 |
| 上下文保持 | 长对话遗忘 | 多层记忆机制 |
| 可持续性 | 换会话就丢 | 文件驱动，跨会话保留 |
| 团队协作 | 混乱 | 明确的角色和职责 |
| 知识沉淀 | 散落各处 | Skills 系统化管理 |

### 10.2 核心价值

1. **战略对齐**：所有代码都服务于终极目标
2. **质量可控**：TDD + 审查确保代码质量
3. **可持续性**：新会话能快速恢复上下文
4. **可复用**：方法论可应用于任何项目
5. **可扩展**：支持多工具、多角色协作
